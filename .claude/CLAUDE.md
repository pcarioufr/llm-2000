# LLM-2000

Use the name "LLM-2000" consistently in all code, documentation, and UI elements.

## Configuration
- Environment Variables tracked in the .env file used in compose.yml:
  - `DD_ENV`: Datadog environment
  - `DD_VERSION`: Application version
  - `DD_SITE`: Datadog site
  - `DD_TAGS`: Datadog tags to add to the agent
  - `DD_CLIENT_TOKEN`: RUM client token
  - `NOTIF_EMAIL`: Email address to send Datadog notifications to
  - `DD_APPLICATION_ID`: RUM application ID
  - `DD_API_KEY`: Datadog API key [sensitive]
  - `DD_APP_KEY`: Datadog application key [sensitive]
  - `DD_CLIENT_TOKEN`: Datadog client token [sensitive]
  - `DATADOG_SECRET_ACCESS_KEY`: Datadog secret access key for Synthetics worker [sensitive]
  - `DATADOG_PUBLIC_KEY_PEM`: Datadog public key for Synthetics worker [sensitive]
  - `DATADOG_PRIVATE_KEY`: Datadog private key for Synthetics worker [sensitive]
  - `DD_OP_PIPELINE_ID`: Observability Pipelines Worker pipeline ID [auto-generated]

- Generated Environment Files (via terraform.sh):
  - `.env/datadog-synthetics-worker.env`: Synthetics private location credentials
  - `.env/rum.env`: RUM application configuration
  - `.env/datadog-op-worker.env`: Observability Pipelines Worker configuration
  - These files are auto-generated by running `./terraform.sh apply`

- Default Files:
  - `flask/default_prompt.txt`: Default system prompt for new chats, can be overridden per user through the UI

## Common Debugging Tips
- Redis data inspection:
  - Chat history: `HGETALL chat:{user_id}`
  - System prompt: `GET prompt:{user_id}`
- Stream processing:
  - Check browser console for token processing logs
  - SSE connections visible in Network tab
- CSS debugging:
  - Button sizes: header buttons (32px), main buttons (40px)
  - Color variables in variables.css for theming

## Backend Components
- `app`: Main Flask application (flask/app/__init__.py)
- `ChatService`: Core chat logic and Redis operations (flask/app/services/chat_service.py)

## API Routes (flask/app/api/routes.py)
- GET `/api/chat`: Check chat existence and load history
- POST `/api/chat`: Send message and get streaming response
- DELETE `/api/chat`: Clear chat history
- GET `/api/chat/welcome`: Get streaming welcome message
- GET `/api/prompt`: Load system prompt
- POST `/api/prompt`: Save system prompt

## Dependencies
- Redis: Chat history and prompt storage
  - Connection: REDIS_URL environment variable
  - Data structure: Hash for chat history, String for system prompt
  - Keys: `chat:{user_id}` for history, `prompt:{user_id}` for system prompt

- Datadog: Monitoring and tracing
  - APM tracing on all routes
  - RUM monitoring in frontend
  - Log collection from both frontend and backend
  - Observability Pipelines Worker sits between Datadog Agent and Datadog Cloud
    - Location: datadog-op-worker/
    - Configuration: Managed in Datadog via Terraform (remote configuration mode)
    - Terraform: terraform/conf/observability-pipeline.tf [source of truth for pipeline config]
    - Worker fetches configuration from Datadog using DD_OP_PIPELINE_ID
    - Receives metrics (port 8282), logs (port 10514), and traces (port 8126) from Agent
    - Forwards to Datadog Cloud with optional transformations
    - Activation requires setting `DD_OBSERVABILITY_PIPELINES_WORKER_LOGS_ENABLED=true` in `.env/datadog.env`

## Terraform Infrastructure
- Location: terraform/conf/
- Managed Resources: Dashboards, Monitors, RUM Application, Synthetics Tests, Observability Pipelines
- Usage:
  - Initialize: `./terraform.sh init`
  - Plan: `./terraform.sh plan`
  - Apply: `./terraform.sh apply` (also generates .env files)
  - Destroy: `./terraform.sh destroy`
